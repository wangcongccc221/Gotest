## Project Overview

HarmonyOS (鸿蒙) fruit sorting application with:
- **Frontend**: ArkTS/HarmonyOS UI (entry/src/main/ets)
- **Backend**: Go 1.25 compiled to native shared library (go/ohos)
- **Bridge**: C++ NAPI layer (entry/src/main/cpp)
- **Database**: SQLite via GORM
- **Communication**: TCP/CTCP, WebSocket, Modbus TCP, OPC UA

The Go backend runs as `libohos.so` inside the HarmonyOS app process, exposing functions through C ABI to the ArkTS frontend.

## Build Commands

### Build Go Backend
```powershell
# Rebuild Go shared library (required after Go code changes)
.\go\ohos\build_ohos.ps1

# Output: entry/libs/arm64-v8a/libohos.so
# Also generates: entry/src/main/cpp/include/libohos.h
```

Requirements:
- HarmonyOS SDK at `$env:OHOS_SDK` (default: `D:\harmSdk\command-line-tools\sdk\default\openharmony`)
- Go 1.25+
- Builds for `arm64-v8a` architecture using LLVM from HarmonyOS SDK

### Build HarmonyOS App
Use DevEco Studio or hvigor to build the full app. The native module (libentry.so) links against libohos.so automatically via CMake.

### Test Go Backend
```powershell
cd go\ohos
go test ./...
```

## Architecture

### Call Chain
```
ArkTS Page (*.ets)
  ↓ import libentry.so
C++ NAPI (napi_init.cpp)
  ↓ calls Go exports
Go Backend (main.go, //export functions)
  ↓ business logic
Go Modules (Tcp/, database/, modbus/, opcua/)
```

### Go Module Structure (`go/ohos/`)
- **main.go**: C ABI exports (`//export GoStartServer`, etc.)
- **Tcp/**: CTCP server/client, WebSocket bridge, data parsing
  - `ctcpserver.go`: CTCP listener, command dispatch (0x1000=StGlobal, 0x1001=StStatistics)
  - `websocket.go`: WebSocket hub, frontend ↔ backend messaging
  - `ctcp_structs.go`: StGlobal, StStatistics, data structure definitions
  - `realtime_save*.go`: 3-second aggregation, database write scheduling
- **database/**: GORM models, migrations, REST API routes
  - `orm.go`: initialization, route registration
  - `tb_*.go`: table models (fruit_info, grade_info, export_info, etc.)
- **modbus/**: Modbus TCP server
- **opcua/**: OPC UA server

### Data Flow
1. **Initialization**: ArkTS calls `GoInitORMWithPath()`, `GoStartTCPServer()`
2. **Configuration sync**:
   - Frontend WebSocket connects to `ws://127.0.0.1:18080/ws/data`
   - Frontend sends `{"type":"requestStGlobal"}`
   - Go backend sends CTCP DISPLAY_ON/SYNC to FSM
   - FSM responds with 0x1000 (StGlobal config)
   - Go parses → JSON → WebSocket publish to frontend
3. **Real-time stats**: FSM sends 0x1001 (StStatistics) periodically → Go parses → WebSocket push
4. **Database writes**: Go aggregates stats every 3 seconds, writes to SQLite via GORM

### Frontend Structure
- **Entry point**: `entry/src/main/ets/pages/Index.ets`
- Main pages registered in `entry/src/main/resources/base/profile/main_pages.json`
- Native bridge: `entry/src/main/cpp/napi_init.cpp` wraps Go functions for ArkTS

## Development Workflow

### After Modifying Go Code
1. Run `.\go\ohos\build_ohos.ps1` to rebuild libohos.so
2. Rebuild HarmonyOS app in DevEco Studio (native module will link new .so)
3. Redeploy to device/emulator

### After Adding New Go Export
1. Add `//export GoFunctionName` in `go/ohos/main.go`
2. Rebuild with `build_ohos.ps1` (generates updated libohos.h)
3. Add C++ wrapper in `entry/src/main/cpp/napi_init.cpp`
4. Register in `napi_property_descriptor` array
5. Call from ArkTS via `import testNapi from 'libentry.so'`

### WebSocket Protocol
Backend sends frames as:
```json
{
  "type": "data",
  "topic": "stglobal" | "statistics" | "grade",
  "data": { /* JSON object */ },
  "at": 1710000000000
}
```

Frontend topics:
- `stglobal`: full config (StGlobal structure)
- `statistics`: real-time counters (StStatistics structure)
- `grade`: grade/weight settings

### CTCP Command IDs
- `0x1000`: FSM config (StGlobal) — full device configuration
- `0x1001`: FSM statistics (StStatistics) — real-time sorting counters
- `0x1002`: FSM grade info — quality/size grade settings

## Common Patterns

### Go ↔ C ABI
Go exports use C types:
```go
//export GoStartServer
func GoStartServer() C.int {
    return C.int(tcp.StartServer())
}

//export GoFreeCString
func GoFreeCString(value *C.char) {
    C.free(unsafe.Pointer(value))
}
```

String returns must be freed by caller via `GoFreeCString()`.

### WebSocket Publishing (Go Backend)
```go
// Publish JSON to all connected WebSocket clients
PublishWebSocketJSON(webSocketTopicStGlobal, jsonString)
```

### Database Access (Go Backend)
```go
// Get GORM instance
db := getORMDB()
if db == nil {
    return errors.New("database not initialized")
}

// Query
var fruits []TbFruitInfo
db.Where("CompletedState = ?", "completed").Find(&fruits)
```

## Important Notes

- **TLS workaround**: Uses `GOOS=android` with custom TLS wrapper (`ohos_android_tls_wrap.go`) because Go doesn't officially support HarmonyOS. The wrapper redirects Android TLS slots to HarmonyOS pthread keys.
- **liblog.so compat**: `native/log_compat/` provides Android log API shim using HarmonyOS hilog, required for Go android runtime.
- **Thread safety**: WebSocket hub uses channels and goroutines; CTCP server runs in separate goroutine; GORM uses connection pooling.
- **Memory**: Go runtime manages its own heap; C strings returned to ArkTS must be freed via `GoFreeCString()`.
