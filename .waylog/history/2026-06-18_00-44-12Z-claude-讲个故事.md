**《那个永远跑不完的循环》**

有个程序员叫小林，写了个夜间批处理任务，每天凌晨三点跑数据。

上线第一晚，任务正常结束。第二晚，没结束。第三晚，还在跑。

他查日志，发现循环里有这么一行：

```
while (任务未完成) {
    处理下一批();
    // 等数据就绪
}
```

问题是——"处理下一批"会**生成**新的一批。于是任务越跑越多，像一个永远追不上自己影子的人。

小林盯着屏幕看了很久，最后加了一行注释，没改代码，下班了。

注释是这么写的：

```
// 它和我一样，不知道什么时候才算做完。
```

第二天产品经理看到日志炸了，跑来问他怎么回事。小林指了指那行注释说："你看，它至少很努力。"

后来这个 bug 修好了，但那行注释被同事们郑重地保留了下来，成了团队代码库里一处小小的、心照不宣的纪念碑。

## 一、把 `0505.sql` 转成 SQLite 能吃的脚本

`0505.sql` 不能直接喂给 SQLite。我实测了它里面的内容，这些是 SQLite **吃不下**的：

| MySQL 专有 | 数量 | SQLite 怎么办 |
|---|---|---|
| 头部 `/*!40101 SET... */`、`SET NAMES`、`SET TIME_ZONE` | 一片 | 全删 |
| `CREATE DATABASE` / `USE \`db_fruitsort\`` | 2 句 | 全删（SQLite 一个文件就是一个库）|
| **存储过程** `CREATE PROCEDURE`+`DELIMITER` | **6 个**（22 个 DELIMITER）| **全删**，见下方说明 |
| 表尾 `ENGINE=InnoDB CHARSET=utf8mb4 COLLATE=...` | 89 个表 | 删成 `);` |
| `FID int NOT NULL AUTO_INCREMENT` + `PRIMARY KEY(FID)` | 每表 | 改成 `FID INTEGER PRIMARY KEY AUTOINCREMENT` |
| 列上的 `CHARACTER SET utf8mb4 COLLATE ...` | 大量 | 删掉 |
| 数据里 `\'` 转义 | INSERT 块 | 改成 `''`（两个单引号）|

**最关键的两个坑：**

1. **那 6 个存储过程别想搬过去**——SQLite 根本没有存储过程。它们是 `AlterFruitProcessTables`、`CreateMonthsTableOfPackingInfo`、`NextSequence` 这类"按年/月动态建表、取序列号"的逻辑。在 SQLite 方案里这些活儿是**程序用 C++ 干的**（项目里的 `RSS/Patch/` 补丁体系和 `createTable()` 就是干这个的）。所以转换时**直接丢弃存储过程**，功能不丢。

2. **`int AUTO_INCREMENT` 主键**：MySQL 那套 `int NOT NULL AUTO_INCREMENT, PRIMARY KEY(FID)` 直接进 SQLite 不会自增。SQLite 必须是 `INTEGER PRIMARY KEY AUTOINCREMENT`（类型得是 `INTEGER`，关键字得是 `AUTOINCREMENT`）。

### 怎么转（按省心程度排）

- **最省心 —— 用图形工具的"数据传输"**：装 **DBeaver**（免费）或 Navicat，连上你那台 MySQL（127.0.0.1 / `db_fruitsort`），用 **Export/数据传输** 目标选 SQLite，它会逐表按 SQLite 语法重建并搬数据，自增、类型、转义都自动处理。存储过程会被跳过，正好。
- **命令行脚本** —— 开源的 `mysql2sqlite`（awk 脚本，GitHub 上搜），专门转 mysqldump，能处理 AUTO_INCREMENT/ENGINE/反引号；但要先把 6 个存储过程段手动删掉再喂给它。
- **纯手工 sed** —— 89 个表，容易漏，不推荐。
