**OpenAI Codex**

“产品名称”这个名字确实有点误导，它不是一个真正保存的“产品档案名”。

它实际意思是：**某个出口当前绑定了哪些等级/规格的果子，系统动态拼出来的一串说明文字。**

举个例子：

你在等级表里配置了：

- 尺寸：`70`、`80`
- 品质：`一级`、`特级`

然后你勾选：

- `70 + 一级` 掉到出口 3
- `80 + 特级` 也掉到出口 3

那出口 3 的“产品名称”就会动态显示成类似：

```text
70.一级, 80.特级
```

所以它更准确的名字应该叫：

```text
出口当前勾选的等级名称
```

它和另外两个名称的区别是：

- **自定义名称**：你给出口起的内部名字，存在 `StExitInfos.ExitName`
- **显示名称**：你手动输入给外部大屏看的名字，存在 `出口X显示名称`
- **产品名称**：系统根据“哪些等级会掉到这个出口”实时算出来，不存数据库

外部大屏那里有个开关：

- `displayType` 某出口 bit = `0`：大屏显示“产品名称”，比如 `70.一级`
- `displayType` 某出口 bit = `1`：大屏显示你手填的“显示名称”，比如 `精品线A`

所以“产品名称”本质上就是：**不手写，自动根据出口勾选关系生成的出口内容说明。**


后面还有：

```text
49,51,52,51,50  => 13432
49,50,51,50,49,51 => 123213
```

所以 WebSocket 回来的 `ExitName` 里确实有 3 个名字。

但是你贴的数据库这段：

```json
"ExitInfos":"AAAAAAAAAAAAAAAAMTM0MzIA..."
```

我解了一下，开头是：

```text
出口1: 空
出口2: 13432
出口3: 空
出口4: 123213
```

它**没有 `56666`**。如果数据库里是当前这次保存后的最新值，Base64 应该以这个开头：

```text
NTY2NjYAAAAAAAAAMTM0MzIA...
```

因为 `NTY2NjY` 就是 `56666` 的 Base64。

所以结论是：

1. **这次前端已经发了。**
2. **后端也回推了带 `56666 / 13432 / 123213` 的数据。**
3. 你现在看的这条数据库值，可能不是最新那条 `ExitInfos`，或者看的不是 Go 后端实际连接的那个数据库文件，或者复制的值不是完整最新值。

```sql
SELECT FID, FModuleName, FType, FValue
FROM tb_Sys_Configs
WHERE FModuleName = 'RSS'

ORDER BY FID DESC
LIMIT 5;
```

重点看 `FID` 最大的那条。Go 代码读取/保存都是按 `FID desc` 找最新的 `FType='ExitInfos'`。
