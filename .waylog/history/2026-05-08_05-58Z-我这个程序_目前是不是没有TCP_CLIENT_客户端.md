我按 48 源码梳理出来了。这里的“握手”不是 TCP 三次握手，而是它自己的应用层协议头。

**1. 上位机客户端发给下位机的内容**

在 [ctcpclient.cpp](<C:/Users/AI008/Desktop/源文件/48/RSS/Base/ctcpclient.cpp:218>)：

```cpp
SendCMD cmd;
cmd.SYNC = 0x434e5953;
cmd.nSrcId = ConstPreDefine::HC_ID;
cmd.nDestId = nDestId;
cmd.nCmd = nCmd;
bytes.append((char*)&cmd, sizeof(SendCMD));
socket->Send(bytes);
socket->Send(data);
```

所以发送格式是：

```text
SYNC + nSrcId + nDestId + nCmd + data
```

具体是 16 字节头：

```text
4字节  SYNC    = "SYNC"
4字节  nSrcId  = HC_ID = 0x1000
4字节  nDestId = 目标下位机 ID
4字节  nCmd    = 命令 ID
后面   data    = 可选参数数据
```

`SYNC = 0x434e5953` 在小端机器里发出去就是 ASCII：

```text
53 59 4E 43 = "SYNC"
```

启动时 48 这里会发：

[main.cpp](<C:/Users/AI008/Desktop/源文件/48/RSS/main.cpp:131>)

```cpp
GlobalDataInterface::TransmitParam(-1, HC_FSM_COMMAND_TYPE::HC_CMD_DISPLAY_ON, QVariant(QVariant::Int));
```

也就是给下位机发 `HC_CMD_DISPLAY_ON`。这个命令本身不带 data，所以实际就是 16 字节头。

**2. 上位机客户端连下位机哪个端口**

在 `SyncRequest` 里按 `nCmd` 判断端口：

```text
FSM 命令：端口 1279
IPM 命令：端口 1289
WAM 命令：端口 1299
SIM 命令：端口 1129
ACS 命令：端口 4127
```

这些常量在 [interface.h](<C:/Users/AI008/Desktop/源文件/48/RSS/Base/interface.h:6779>)：

```cpp
FSM_PORT_NUM = 1279
IPM_PORT_NUM = 1289
SIM_PORT_NUM = 1129
WAM_PORT_NUM = 1299
ACS_PORT_NUM = 4127
```

IP 默认是：

```cpp
LC_IP_ADDR_TEMPLATE = "192.168.0."
```

然后拼接子系统 ID。比如 `EncodeSubsys(0)` 是 `0x0100`，`GetSubSysID(0x0100)` 得到 `16`，所以 FSM 地址就是：

```text
192.168.0.16:1279
```

**3. 下位机返回数据发到哪里**

不是返回到这个 TCP client 连接里。

48 的 `CTcpClient` 发完就：

```cpp
socket->DestroySocket();
delete socket;
```

它没有在这个 client 上读返回数据。

下位机返回数据时，是下位机作为 TCP client，主动连上位机开的 TCP server。

上位机在 [globaldatainterface.cpp](<C:/Users/AI008/Desktop/源文件/48/RSS/Base/globaldatainterface.cpp:477>) 启动了两个服务器：

```cpp
m_TcpSvrImage.Start(HC_IP_ADDR, HC_PORT1_NUM, HC_ID, ...)
m_TcpSvrStat.Start(HC_IP_ADDR, HC_PORT2_NUM, HC_ID, ...)
```

对应端口：

```text
上位机 IP: 192.168.0.15
图像服务器: 1127
状态/配置服务器: 1128
```

所以方向是：

```text
上位机 CTcpClient -> 下位机端口 1279/1289/1299...
下位机再主动连接 -> 上位机 TcpServer 1127/1128
```

**4. 下位机回上位机的数据格式**

上位机服务器也要求同样的头格式。

在 [tcpserver.cpp](<C:/Users/AI008/Desktop/源文件/48/RDP_Communication/Tcp/tcpserver.cpp:131>)：

```cpp
先读 4 字节，必须是 "SYNC"
再读 12 字节：
nSrcId + nDstId + nCmdId
```

并且 `nDstId` 必须等于上位机 ID：

```cpp
if(nDstId != m_DstId) return false;
```

这里 `m_DstId` 是 `HC_ID = 0x1000`。

所以结论很清楚：

```text
上位机发下位机：
SYNC + HC_ID + 下位机ID + 命令 + data

下位机回上位机：
SYNC + 下位机ID + HC_ID + 命令 + data
```

下位机不是回到原来的短连接 client，而是连上位机服务器：

```text
1127：图像类数据
1128：配置、统计、版本、重量、波形等状态数据
```

你是怎么把字节转换成结构体的？

---


**OpenAI Codex**

不是用 Go 的 `binary.Read` 一口气转成结构体，而是手动按 48 的 C++ 字段顺序一点点读字节。

比如收到 `FSM_CMD_CONFIG` 后，`payload` 是 `28712` 个字节。解析流程大概是：

```go
sysConfig, err := parseStGlobalSysConfig(payload)
gradeInfo, err := parseStGlobalGradeInfo(payload, sysConfig)
```

核心方式是维护一个 `offset`：

```go
offset := 0
```

每读一个字段，就从 `payload[offset:offset+n]` 取对应字节，然后 `offset += n`。

例如 `int` 是 4 字节，小端序：

```go
value := int32(binary.LittleEndian.Uint32(payload[offset : offset+4]))
offset += 4
```

`short / quint16` 是 2 字节：

```go
value := binary.LittleEndian.Uint16(payload[offset : offset+2])
offset += 2
```

`quint8` 就是 1 字节：

```go
value := payload[offset]
offset++
```

`float` 也是 4 字节，但要先按 `uint32` 读，再转成 `float32`：

```go
value := math.Float32frombits(binary.LittleEndian.Uint32(payload[offset : offset+4]))
offset += 4
```

数组就是循环读，或者直接 `copy`：

```go
copy(sysConfig.NChannelInfo[:], payload[offset:offset+4])
offset += 4
```

为什么不用“一键转 struct”？因为 C++ 结构体里有：

```cpp
quint8
quint16
int
float
ulong
#pragma pack(4)
#if defined(...)
```

这些会涉及内存对齐、条件编译、平台差异。Go 的 struct 内存布局不一定和 C++ 完全一样，直接转很容易错。所以现在是按协议字节布局手动解析，稳一点。

但是你不知道嵌套结构体他具体的偏移是多少啊 为什么不用unsafe 包处理二进制 我有一个问题

---


**OpenAI Codex**

你的问题是对的：嵌套结构体的偏移不能靠猜。

但这里不适合直接用 Go 的 `unsafe` 把 `[]byte` 强转成 struct，原因是：

1. **Go struct 布局不等于 C++ struct 布局**
   48 里有 `quint8 / quint16 / int / short / float / ulong`，还有 `#pragma pack(4)`。Go 的字段对齐规则和 C++ 不一定一致。

2. **`ulong` 最危险**
   C++ 里的：

   ```cpp
   ulong exit;
   ```

   在不同平台可能是 4 字节，也可能是 8 字节。你这次日志已经证明这里必须按 8 字节，不然 `StGradeInfo` 后面全错位。

3. **`#if defined(...)` 会改变结构体大小**
   比如：

   ```cpp
   #if defined(LS8)||defined (MAX64)
       ushort nIQSEnable;
   #else
       quint8 nIQSEnable;
   #endif
   ```

   如果 Go 这边不知道下位机编译时开了哪个宏，`unsafe` 直接转会更危险。

4. **字节序问题**
   TCP 传来的二进制是小端数据。`unsafe` 是按当前 CPU 内存布局解释，不显式处理字节序。现在用：

   ```go
   binary.LittleEndian.Uint32(...)
   ```

   至少每个字段怎么来的很明确。

5. **`unsafe` 只能在布局 100% 一样时用**
   如果偏移错 1 个字节，整个结构体后面全错，而且很难查。

现在这种方式本质上是“按 48 的 C++ 结构体顺序手写解包”。嵌套结构体的偏移确实要靠：

```text
前一个字段大小 + C++ 对齐规则 + 条件编译字段
```

来算。更稳的做法是后面专门建一个 `ctcp_parser.go` 或 `ctcp_layout.go`，把每个结构体的 `Size` 和关键字段 offset 写清楚。这样不会散在 server 逻辑里。

所以结论是：`unsafe` 可以用，但现在不该用。我们现在连 `StGlobal` 的完整 C++ 布局还没完全确认，直接 `unsafe` 会把问题藏起来，不会解决问题。
