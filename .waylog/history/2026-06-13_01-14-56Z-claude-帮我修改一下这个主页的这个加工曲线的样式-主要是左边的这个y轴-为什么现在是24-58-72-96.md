**Claude**

改好了。主页的"加工曲线"用的是 [RealTimeProcessingChart.ets](../new/my_harmony/entry/src/main/ets/components/charts/RealTimeProcessingChart.ets)（已在 [DataTablesTabBar.ets:1810](../new/my_harmony/entry/src/main/ets/pages/home/DataTablesTabBar.ets#L1810) 确认就是主页这块图表）。

**原因**：这个图表的 Y 轴之前没有指定范围，mccharts 会根据数据自动取整到 120，再除以默认的 5 个分段，就得到了 24/48/72/96/120 这种奇怪的刻度。

**修改**：在 [RealTimeProcessingChart.ets:85-90](../new/my_harmony/entry/src/main/ets/components/charts/RealTimeProcessingChart.ets#L85-L90) 的 `yAxis` 里固定了范围：

```ts
min: 0,
max: 100,
interval: 20,
```

由于三条曲线（运行速度/分选效率/实时产量）本身就是归一化到 0~100% 的，固定上限 100 不会截断数据。现在左侧刻度会显示 0% / 20% / 40% / 60% / 80% / 100%。

另外提一句：弹窗里用的另一个图表组件 [ProcessingCurveChart.ets:54-55](../new/my_harmony/entry/src/main/ets/components/charts/ProcessingCurveChart.ets#L54-L55) 也写死了 `max: 120, interval: 24`，按你说的只改主页这块，那个我没动；如果也要改成一样的，跟我说一声即可。

**Claude**

已经撤销了，`git diff` 确认 [RealTimeProcessingChart.ets](../new/my_harmony/entry/src/main/ets/components/charts/RealTimeProcessingChart.ets) 和提交版本完全一致，恢复原样。

**为什么会挤成一团、横线消失** —— 这不是改法的问题，而是项目用的图表库 mccharts 2.8.9 自身的 bug。在 [axis.ts:517](../new/my_harmony/oh_modules/@mcui/mccharts/src/main/ets/mcCore/core/axis.ts#L517) 里有一行条件写反了：

```js
const ratio = (maxValue - minValue) === 0 ? (abs(gapLength) / (maxValue - minValue)) : 0;
```

正常应该是"差值不为 0 时才计算比例"，它写反了，导致只要给 Y 轴手动设置了 `min`/`max`，所有刻度的纵坐标都会乘 0，全部叠到同一个点上（就是你看到的挤成一团），横向分割线也跟着叠没了。不设 `min`/`max` 时走的是另一条均分逻辑，所以原来是正常的。

**为什么现在显示 24/48/72/96/120**：不设最大值时，库会自动拿数据峰值（100）上浮 20% 得到 120，再均分 5 段，所以是 24 的倍数。

要改成 0/20/40/60/80/100，目前只有一条路：把库里那行写反的条件修正（一行改动），然后再给 Y 轴设 `min: 0, max: 100, interval: 20`。但有个注意点：`oh_modules` 目录被 git 忽略，直接改库文件的话，下次重新 `ohpm install` 会被覆盖，需要重装后再补一次（或者把改过的库文件拷一份到项目里做记录）。
