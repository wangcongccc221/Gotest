**1. 等级设置页的“自动生成”**  
入口是 [levelform.cpp](E:/new/48/RSS/Forms/levelform.cpp:1011)。

它不是重新算等级数量，而是按已有的 `nSizeGradeNum` 遍历：

```cpp
for (int i = 0; i < tempGradeInfo.nSizeGradeNum; i++)
```

逻辑是：

- 等级名来自重量表：`GradeWeightlistViewEx` 第 0 行、`i * 2 + 1` 列。
- 写入 `strSizeGradeName[i]`。
- 分界值写入每个品质组合下的 `grades[j * MAX_SIZE_GRADE_NUM + i].nMinSize`。
- `j` 的范围是品质数量：
  - 如果 `nQualityGradeNum > 0`，用它。
  - 否则按 1 个品质处理。
- 最后一个等级的 `nMinSize` 固定写 `0.0`。

所以普通等级自动生成的范围是：

```text
尺寸等级: 0 .. nSizeGradeNum - 1
品质层: 0 .. max(nQualityGradeNum, 1) - 1
索引: qualityIndex * MAX_SIZE_GRADE_NUM + sizeIndex
```

**2. 品质等级页的“自动生成”**  
入口是 [qualityform.cpp](E:/new/48/RSS/Forms/qualityform.cpp:1992)。

它会根据启用的品质维度做笛卡尔积：颜色、形状、瑕疵、碰伤、腐烂、糖度、酸度、空心、果皮、褐变、糖心、硬度、水分等。

核心规则：

- 每个启用的维度，用对应的等级数量参与相乘。
- 未启用的维度，范围按 1，并且最终写 `0x7F`，表示“全部/不限制”。
- 颜色有特殊逻辑：
  - 如果是平均值模式，根据 `ColorIntervals[0]` 自动用 2 或 3 个等级。
  - 如果是百分比模式，用 `ColorGradeNum`。
- 总品质等级数 `TotalGradeNum` 超过 16 就直接提示并返回。
- 生成出的名称是各维度等级名拼接，比如 `红.圆.无瑕`，没有名称时用“新等级”。

保存时在 [qualityform.cpp](E:/new/48/RSS/Forms/qualityform.cpp:1337)：

- `tempGradeInfo.nQualityGradeNum = Quality_GradeInfo.GradeCnt`
- 品质名写入 `strQualityGradeName`
- 每个品质会复制到所有尺寸等级：

```text
索引 = qualityIndex * MAX_SIZE_GRADE_NUM + sizeIndex
```

如果品质数量变化，会把所有 `grades[].exit` 清零，防止旧出口映射残留。

我注意到 48 原代码有两个很可疑的点：

- [qualityform.cpp](E:/new/48/RSS/Forms/qualityform.cpp:2150) 腐烂等级那里是 `TotalGradeNum = temp_RotGradeNum`，不是 `*= temp_RotGradeNum`。这会覆盖前面已经乘出来的总数，像是老代码 bug。
- [qualityform.cpp](E:/new/48/RSS/Forms/qualityform.cpp:2496) 拼等级名时 `strG12Name` 出现了两次，也像是 typo。

**3. 结束加工里的“自动生成表格”**  
入口是 [mainform.cpp](E:/new/48/RSS/Forms/mainform.cpp:2004)。

这个不是等级配置自动生成，而是结束加工后生成批次数据/报表。它的等级记录生成在 [globaldatainterface.cpp](E:/new/48/RSS/Base/globaldatainterface.cpp:1745)：

```text
品质数量 = QualityGradeSum > 0 ? QualityGradeSum : 1
尺寸/重量数量 = WeightOrSizeGradeSum > 0 ? WeightOrSizeGradeSum : 1
生成范围 = 品质数量 × 尺寸/重量数量
索引 = qualityIndex * MAX_SIZE_GRADE_NUM + sizeIndex
```

出口记录范围是 `ExportSum`，子系统记录范围是 `SubSystemNum`。

结论：如果你现在要复刻“品质自动生成”，重点不是只改 `nQualityGradeNum`，而是按 48 的逻辑生成 `Quality_GradeInfo.Item[]`，再同步写进 `StGradeInfo.strQualityGradeName` 和 `StGradeInfo.grades[quality * MAX_SIZE_GRADE_NUM + size]`。

**OpenAI Codex**

“实体”就是一张数据库表在代码里的对象表示。

比如 48 里有表 `tb_FruitInfo`，对应 C++ 里这个类：

```cpp
class tb_FruitInfo : public QObject
{
    Q_PROPERTY(int CustomerID MEMBER CustomerID)
    Q_PROPERTY(QString CustomerName MEMBER CustomerName)
    Q_PROPERTY(int BatchWeight MEMBER BatchWeight)
}
```

这就是实体类。它告诉程序：

`tb_FruitInfo` 这张表有哪些字段、字段叫什么、字段是什么类型。

我们 Go 后端里写的这个也是实体：

```go
type TbFruitInfo struct {
    CustomerID   int
    CustomerName string
    BatchWeight  int
}
```

ORM 会根据这个实体去创建表、读表、写表。

简单说：

```text
数据库表 = 真实存数据的表
实体 = 代码里代表这张表的结构体/类
一行数据 = 一个实体对象
```

比如数据库里一条批次记录：

```text
CustomerID = 1
CustomerName = 张三
BatchWeight = 5000
```

在代码里就是一个 `TbFruitInfo` 对象。
