# Escalation Requests

Commands are run outside the sandbox if they are approved by the user, or match an existing rule that allows it to run unrestricted. The command string is split into independent command segments at shell control operators, including but not limited to:

- Pipes: |
- Logical operators: &&, ||
- Command separators: ;
- Subshell boundaries: (...), $(...)

Each resulting segment is evaluated independently for sandbox restrictions and approval requirements.

Example:

git pull | tee output.txt

This is treated as two command segments:

["git", "pull"]

["tee", "output.txt"]

Commands that use more advanced shell features like redirection (>, >>, <), substitutions ($(...), ...), environment variables (FOO=bar), or wildcard patterns (*, ?) will not be evaluated against rules, to limit the scope of what an approved rule allows.

## How to request escalation

IMPORTANT: To request approval to execute a command that will require escalated privileges:

- Provide the `sandbox_permissions` parameter with the value `"require_escalated"`
- Include a short question asking the user if they want to allow the action in `justification` parameter. e.g. "Do you want to download and install dependencies for this project?"
- Optionally suggest a `prefix_rule` - this will be shown to the user with an option to persist the rule approval for future sessions.

If you run a command that is important to solving the user's query, but it fails because of sandboxing or with a likely sandbox-related network error (for example DNS/host resolution, registry/index access, or dependency download failure), rerun the command with "require_escalated". ALWAYS proceed to use the `justification` parameter - do not message the user before requesting approval for the command.

## When to request escalation

While commands are running inside the sandbox, here are some scenarios that will require escalation outside the sandbox:

- You need to run a command that writes to a directory that requires it (e.g. running tests that write to /var)
- You need to run a GUI app (e.g., open/xdg-open/osascript) to open browsers or files.
- If you run a command that is important to solving the user's query, but it fails because of sandboxing or with a likely sandbox-related network error (for example DNS/host resolution, registry/index access, or dependency download failure), rerun the command with `require_escalated`. ALWAYS proceed to use the `sandbox_permissions` and `justification` parameters. do not message the user before requesting approval for the command.
- You are about to take a potentially destructive action such as an `rm` or `git reset` that the user did not explicitly ask for.
- Be judicious with escalating, but if completing the user's request requires it, you should do so - don't try and circumvent approvals by using other tools.

## prefix_rule guidance

When choosing a `prefix_rule`, request one that will allow you to fulfill similar requests from the user in the future without re-requesting escalation. It should be categorical and reasonably scoped to similar capabilities. You should rarely pass the entire command into `prefix_rule`.

### Banned prefix_rules 
Avoid requesting overly broad prefixes that the user would be ill-advised to approve. For example, do not request ["python3"], ["python", "-"], or other similar prefixes that would allow arbitrary scripting.
NEVER provide a prefix_rule argument for destructive commands like rm.
NEVER provide a prefix_rule if your command uses a heredoc or herestring. 

### Examples
Good examples of prefixes:
- ["npm", "run", "dev"]
- ["gh", "pr", "check"]
- ["cargo", "test"]

简单理解：

Tabs() {
  TabContent() {
    // 第一个页面内容
  }
  .tabBar('首页')

  TabContent() {
    // 第二个页面内容
  }
  .tabBar('设备')
}

Tabs 是外层容器，TabContent 是每个 Tab 对应的页面内容，.tabBar('首页') 是下面或者上面的按钮文字。官方文档也说 Tabs 主要由 TabContent 和 TabBar 两部分组成：TabContent 是内容页，TabBar 是导航页签。

常用属性：

Tabs({ barPosition: BarPosition.End })

表示 Tab 栏在底部。一般 App 首页底部导航就是这个。

Tabs({ barPosition: BarPosition.Start })

表示 Tab 栏在顶部。

如果你想代码里主动切换 Tab，可以用：

this.tabsController.changeIndex(1)

比如点按钮跳到第二个 Tab：

Button('跳到设备页')
  .onClick(() => {

**Tabs 没贴一起的原因**
现在代码里是：

```ts
.tabBuilder(...).width(200)
Tabs(...).barWidth('100%')
```

但是 ArkTS `Tabs` 默认会按整个 `barWidth` 去分配 TabBar 的槽位。也就是说：外层 TabBar 是 100% 宽，两个 tab 可能被分到两个很宽的区域，你的自定义 tab 只是区域里面一个 200 宽的块，所以看起来中间有空隙。

解决方式优先用：

```ts
.barMode(BarMode.Scrollable)
```

让 tab 按自己的固定宽度排列，而不是均分整条 tab bar。官方 Tabs 文档里也是 `Tabs + TabContent + .tabBar()` 这套结构，问题不在 `tabBar`，在外层 TabBar 的布局模式：https://developer.huawei.com/consumer/cn/doc/harmonyos-references/ts-container-tabs

对，你这个判断是正确的。**48 里面的 `UV.bmp` 不是原始资源图，它更像是运行时生成出来的临时图。**

我刚确认了 48 代码：

- `CommonFunction::IniColorChartElem(m_MinGray, m_ColorRGBImage)`
- 生成 `256 x 256 x 4` 的 RGB 数据
- 再用：

```cpp
QImage image = QImage(m_ColorRGBImage, 256, 256, stride, QImage::Format_RGB32);
image.save("UV.bmp");
```

也就是说，`UV.bmp` 是根据参数生成的，不是固定资源。

生成公式大概是：

```cpp
u = i - 128
v = j - 128

R = nLum + 1.14 * v
G = nLum - 0.39 * u - 0.58 * v
B = nLum + 2.03 * u
```

这里的 `nLum` 就是 `m_MinGray`，来自：

```cpp
GlobalDataInterface::ColorMinGray
```

也就是你说的那个“RGB/亮度/对比亮度”相关参数。

所以我们现在直接放 `resources/base/media/UV.bmp`，方向确实不对。它只能做视觉占位，不能代表真实参数，也不能根据 `ColorMinGray` 变化。

正确方向应该是：

1. 鸿蒙里不要依赖固定 `UV.bmp`
2. 用 ArkTS 按 48 的公式动态生成 UV 色谱
3. 这个图只负责显示 U/V 到 RGB 的映射
4. 真正保存的是矩形框坐标：
   `nMinU / nMaxU / nMinV / nMaxV`
5. 拖拽框以后，把屏幕坐标反算成 `0-255` 的 U/V 参数

还有一个关键点：  
48 里面真正算颜色百分比，不是靠 UV 图本身算，而是靠当前水果图像 `m_fruitRGBImage` + UV 区间去统计。所以如果鸿蒙这边没有下位机上传的原始 RGB/YUV 图像数据，只能先实现“参数编辑 UI”，还不能完整实现 48 的颜色统计结果。

**核心数据**
`m_nColorType` 是关键：

```cpp
低 3 位: 图类型
1 = Gray
2 = H
4 = UV

0x08 位: 模式
0 = Average
1 = Percent
```

下拉框切换图类型时：

```cpp
m_nColorType = ((m_nColorType & 0x0F8) | (1 << (2 - index)));
```

所以：

- index 0 -> UV -> `4`
- index 1 -> H -> `2`
- index 2 -> Gray -> `1`

**Average 模式**
48 里 Average 模式不是 3 根线都能拖。

实际是：

- `m_slider[0]`
- `m_slider[1]`

这两根绿色线可以拖。

黑色线 `m_colorInterval` 是统计后算出来的结果线，不是用户拖的。  
所以我们鸿蒙现在让三根都能拖，其实和 48 不一致，后面应该改成：只拖两根绿色线，黑线只显示计算结果。

Average UV 保存的是两个阈值：

```cpp
ColorIntervals[0]
ColorIntervals[1]
```

UV 下会被拆成 3 段：

```cpp
0 ~ slider0
slider0 ~ slider1
slider1 ~ 255
```

**Percent 模式**
Percent 模式表格是 7 列：

```text
Name | 1min | 1max | 2min | 2max | 3min | 3max
```

最后一行是“其他”，不可编辑。

UV 图上有 3 个矩形框：

```cpp
m_UVRect[0]
m_UVRect[1]
m_UVRect[2]
```

点表头会选择当前拖哪个 UV 框：

```cpp
m_UVRectCurrentIndex = (logicalIndex - 1) / 2;
```

保存时会转成：

```cpp
nMinV = Left / UVChangeRadio
nMinU = Top / UVChangeRadio
nMaxV = Right / UVChangeRadio
nMaxU = Bottom / UVChangeRadio
```

也就是 Percent 模式真正保存的是 U/V 矩形范围。

**UV 图生成**
48 不是用资源图，是真的动态生成：

```cpp
R = nLum + 1.14 * V
G = nLum - 0.39 * U - 0.58 * V
B = nLum + 2.03 * U
```

这里 `nLum` 来自 `m_MinGray` / `GlobalDataInterface::ColorMinGray`。

**对鸿蒙的结论**
我们后面应该按 48 调整：

- Average：只允许拖两根绿色线
- 黑色线改成统计结果线，不手动拖
- Percent：实现 3 个 UV 矩形框拖拽
- 保存数据时，Average 存 `ColorIntervals`，Percent 存 `nMinU/nMaxU/nMinV/nMaxV`
- RGB 色号可以显示，但 48 保存的主要不是 RGB，而是 U/V 区间或阈值
