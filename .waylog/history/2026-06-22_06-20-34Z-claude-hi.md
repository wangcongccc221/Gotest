2. 但链接时用 `--target=aarch64-linux-ohos` 和鸿蒙的 sysroot
3. 加上 TLS 包装层（`ohos_android_tls_wrap.go`）绕过 Android 特定的 TLS 机制
4. liblog.so 兼容层把 Android log API 转成鸿蒙的 hilog

本质是**借用 Android 的 Go runtime（因为指令集相同），再通过兼容层适配鸿蒙的系统调用差异**。

这就是为什么文档里说这是 "POC"（概念验证）方案——不是原生支持，而是通过兼容层绕过去的。

## 关键区别

| 对比项 | 鸿蒙（TypeScript） | 48（C++/Qt） |
|--------|-------------------|--------------|
| **算法** | **递归** | **14层嵌套循环** |
| **代码行数** | ~40 行 | ~700 行 |
| **可维护性** | ✅ 高（添加维度只需修改数组） | ❌ 低（添加维度需要增加一层循环） |
| **可读性** | ✅ 清晰（函数式风格） | ⚠️ 循环嵌套深 |
| **数据结构** | 结构化（`QualityAutoDimension[]`） | 分散的独立变量 |
| **全部标记** | `0x7F (QUALITY_ANY_GRADE)` | `0x7F` |
| **名称分隔** | `.` (点) | `.` (点) |

## 总结

两个项目的**核心思想完全一致**：
- 通过多个维度的等级数相乘得到总等级数
- 生成所有维度的笛卡尔积组合
- 未启用或只有 1 级的维度标记为 `0x7F`（全部）
- 等级名称用"."拼接

但**实现方式大不相同**：
- **鸿蒙用递归**，代码简洁优雅，易于扩展
- **48 用 14 层嵌套循环**，代码冗长，但更直观（一眼看出有 14 个维度）

鸿蒙的实现是对 48 项目的**重构优化版本**，保持了功能一致性，但提升了代码质量。
