# Escalation Requests

Commands are run outside the sandbox if they are approved by the user, or match an existing rule that allows it to run unrestricted. The command string is split into independent command segments at shell control operators, including but not limited to:

- Pipes: |
- Logical operators: &&, ||
- Command separators: ;
- Subshell boundaries: (...), $(...)

Each resulting segment is evaluated independently for sandbox restrictions and approval requirements.

Example:

git pull | tee output.txt

This is treated as two command segments:

["git", "pull"]

["tee", "output.txt"]

Commands that use more advanced shell features like redirection (>, >>, <), substitutions ($(...), ...), environment variables (FOO=bar), or wildcard patterns (*, ?) will not be evaluated against rules, to limit the scope of what an approved rule allows.

## How to request escalation

IMPORTANT: To request approval to execute a command that will require escalated privileges:

- Provide the `sandbox_permissions` parameter with the value `"require_escalated"`
- Include a short question asking the user if they want to allow the action in `justification` parameter. e.g. "Do you want to download and install dependencies for this project?"
- Optionally suggest a `prefix_rule` - this will be shown to the user with an option to persist the rule approval for future sessions.

If you run a command that is important to solving the user's query, but it fails because of sandboxing or with a likely sandbox-related network error (for example DNS/host resolution, registry/index access, or dependency download failure), rerun the command with "require_escalated". ALWAYS proceed to use the `justification` parameter - do not message the user before requesting approval for the command.

**一个 WebSocket 服务器通常需要这些内容**

1. **路由入口**
   - `GET /ws`
   - HTTP upgrade 到 WebSocket
   - origin 检查
   - 连接失败时返回明确错误

2. **连接生命周期**
   - 建立连接
   - 保存连接对象
   - 读循环 `ReadMessage`
   - 写循环 `WriteMessage`
   - 客户端断开时清理资源
   - 服务关闭时主动 close 所有连接

3. **消息协议**
   最好不要只传裸字符串，建议用 JSON 包一层，例如：

   ```json
   {
     "type": "ping",
     "requestId": "xxx",
     "payload": {}
   }
   ```

   ```json
   {
     "type": "pong",
     "requestId": "xxx",
     "ok": true,
     "payload": {},
     "error": ""
   }
   ```

4. **心跳机制**
   - 客户端定期发 `ping`
   - 服务端回 `pong`
   - 设置 read deadline
   - 超时断开连接
   - 避免死连接一直占资源

5. **并发安全**
   gorilla/websocket 的连接不适合多个 goroutine 同时写。通常做法是：
   - 一个读 goroutine
   - 一个写 goroutine
   - 写消息通过 channel 排队

   如果只是单连接 echo，不需要复杂管理。  
   如果后面要广播、推送状态、设备消息分发，就需要：
   - `clients map[*Client]bool`
   - register/unregister channel
   - broadcast channel
   - 每个 client 独立 send queue

   - `ping` / `pong`：心跳

   - JSON 解析失败
   - 不支持的 message type
   - 业务执行失败
   - 写入失败
   - 客户端异常断开

9. **安全和限制**
   - `CheckOrigin` 不建议一直 `return true`，至少可以先限制本机或设备局域网
   - 设置最大消息大小，比如 64KB 或 1MB

		return nil

			return nil
		}

	case reflect.Struct:
		out := make(map[string]any)
		t := v.Type()
		for i := 0; i < v.NumField(); i++ {

			out[sf.Name] = valueToReflectJSON(v.Field(i))

		}

		}

对，本质上不是“全局变量”的问题，而是 Go 的**导出规则**。

在 Go 里：

- 大写开头的标识符：导出，包外可见
- 小写开头的标识符：未导出，只在当前 package 内可见

`encoding/json` 是标准库的另一个 package。它用反射读取结构体字段时，只会处理**导出字段**，也就是大写开头的字段。小写字段即使有值，也会被忽略。

例如：

```go
type User struct {
	Name string
	age  int
}

b, _ := json.MarshalIndent(User{Name: "Tom", age: 18}, "", "  ")
fmt.Println(string(b))
```

输出只有：

```json
{
  "Name": "Tom"
}
```

`age` 不会出现。

这不是因为 `age` 不是全局变量，而是因为它是结构体的**未导出字段**。标准库不能像你当前 package 里的普通代码那样直接访问它。

你写的 `valueToReflectJSON` 是自己在同一个 package 里用反射硬取字段值，所以可以把一些小写字段也转出来。但直接 `json.MarshalIndent` 会遵守 Go 的导出规则，只认大写字段。

**OpenAI Codex**

`json:"type"` 不是为了让字段能在其他 package 里用。  
字段能不能被其他 package 访问，靠的是字段名大写，比如：

```go
Type string
```

而 `json:"type"` 控制的是：**转成 JSON 以后，字段名叫什么**。

比如没有 tag：

```go
type Msg struct {
	Type string
}
```

输出是：

```json
{
  "Type": "ping"
}
```

加了 tag：

```go
type Msg struct {
	Type string `json:"type"`
}
```

输出是：

```json
{
  "type": "ping"
}
```

所以两件事分开看：

- `Type`：Go 代码里的字段名，大写开头，表示导出字段
- `json:"type"`：JSON 里的 key 名，控制给前端/客户端看的名字

你的结构体如果直接给 Go 内部用，字段全大写，直接 `json.MarshalIndent` 就行。输出会是：

```json
{
  "NSubsysId": 1,
  "NVersion": 2
}
```

```json
{
  "type": "data",
  "requestId": "xxx",
  "topic": "statistics"

var wsClients = map[*websocket.Conn]bool{}
var wsMu sync.Mutex
var latestJSON string

func handleWebSocket(ctx *gin.Context) {
	conn, err := upgrader.Upgrade(ctx.Writer, ctx.Request, nil)
	if err != nil {
		return
	}
	defer conn.Close()

	wsMu.Lock()
	wsClients[conn] = true
	wsMu.Unlock()

	defer func() {
		wsMu.Lock()
		delete(wsClients, conn)
		wsMu.Unlock()
	}()

	for {
		if _, _, err := conn.ReadMessage(); err != nil {
			return
		}
	}
}

func SendJSONToWebSocket(jsonText string) {
	wsMu.Lock()
	latestJSON = jsonText
	for conn := range wsClients {
		_ = conn.WriteMessage(websocket.TextMessage, []byte(jsonText))
	}
	wsMu.Unlock()
}
```
